# 스킬: Domain Rules (도메인 주도 설계 규칙)

> **목적:** 도메인 주도 설계(DDD) 패턴을 강제하여, AI가 읽어야 할 파일 범위를 좁히고 환각(Hallucination)을 방지한다.
> **적용 시점:** 프로젝트 시작(KICKOFF) 시 최초 작성, 이후 모든 단계에서 참조.

## 🎯 핵심 원칙
- **비즈니스 용어 = 코드 용어 (Ubiquitous Language)**
  - 변수명, 함수명, 파일명, 폴더명은 비즈니스 도메인의 실제 용어와 정확히 일치해야 한다.
  - 예: 결제 도메인이면 `payment/`, 회원이면 `user/` 또는 `member/`
- **도메인 단위 폴더 분리**
  - 기능별이 아닌 도메인별로 폴더를 나눈다.
  - 각 도메인 폴더는 독립적으로 이해할 수 있어야 한다.
- **결합도 최소화, 응집도 최대화**
  - 하나의 도메인 폴더 안에 해당 도메인의 모든 관련 파일(모델, 서비스, 컨트롤러, 테스트)을 모은다.
  - 도메인 간 의존성은 명시적 인터페이스를 통해서만 허용한다.

## 📂 DDD 스캐폴딩 예시
```
src/
├── payment/           # 결제 도메인
│   ├── models/
│   ├── services/
│   ├── controllers/
│   └── tests/
├── user/              # 회원 도메인
│   ├── models/
│   ├── services/
│   ├── controllers/
│   └── tests/
└── shared/            # 공유 유틸리티 (최소한으로 유지)
    ├── utils/
    └── config/
```

## 📋 용어 사전 (프로젝트별 작성)
| 비즈니스 용어 | 코드 용어 | 설명 |
|---|---|---|
| (예: 주문) | `order` | 사용자가 생성한 구매 요청 |
| (예: 배송) | `delivery` | 주문의 물리적 이동 |

> **AI 에이전트 행동 지침:**
> 이 용어 사전에 없는 새로운 도메인 용어를 임의로 만들지 마라.
> 새 용어가 필요하면 사전에 추가하고 `context/2_context_note.md`에 기록해.

## 🚫 금지사항
- 도메인과 무관한 generic 이름 사용 (예: `utils/helper.js`에 비즈니스 로직 넣기)
- 여러 도메인의 로직을 하나의 파일에 혼합
- 용어 사전에 없는 네이밍 임의 사용
- 도메인 폴더 바깥에서 도메인 로직 구현
